# EDINET財務データ抽出ツール 詳細設計書

最終更新: 2026-06-27
対象システム: https://edinet-app-two.vercel.app

---

## 1. システム概要

### 1.1 目的

証券コード（4桁）を入力するだけで、金融庁の開示システム「EDINET」から有価証券報告書（XBRL形式）を自動取得し、売上高・利益・1株指標などの財務データを抽出して画面表示・Excel/CSV/Googleスプレッドシートに出力するWebアプリケーション。単一企業の検索と、最大70社までの一括処理（複数年度取得も可）に対応する。

### 1.2 想定利用者

個別株の財務分析を行う個人投資家・社内の調査担当者。専門知識がなくても証券コードか会社名を入れるだけで使える。

### 1.3 主要機能一覧

| # | 機能 | 概要 |
|---|------|------|
| 1 | 会社名検索 | 会社名の一部入力で証券コード・正式社名をサジェスト |
| 2 | 単一企業検索 | 証券コード1件を指定し、最新（または期間指定）の有価証券報告書から財務データを取得 |
| 3 | 一括処理 | 証券コードを最大70件まとめて入力し、順次データ取得（進捗バー表示） |
| 4 | 市場データ取得 | Yahoo Financeから前日終値・業種を取得し、PER/PBR/配当利回りを実績ベースで算出 |
| 5 | 理論株価算出 | 取得した財務データから簡易理論株価モデルで参考株価を算出 |
| 6 | CSV/Excel出力 | 取得結果一覧をCSVまたはExcel(.xlsx)としてダウンロード |
| 7 | Googleスプレッドシート出力 | 取得結果を新規シートとして既存スプレッドシートに書き込み |

## 5. 処理フロー（全体）

### 5.1 単一企業検索のシーケンス

```
[ユーザー] 証券コード入力 or 会社名検索で選択 → [検索]ボタン
   │
   ▼
GET /api/financials?secCode=XXXX
   │
   ├─① EDINET書類一覧API(documents.json)を直近400日分、30日刻みで
   │    検索しdocTypeCode=120(有価証券報告書)かつ証券コード一致の
   │    最新1件を発見（lib/edinet-client.ts: findDocumentsBySecCode）
   │
   ├─② 対象書類のZIPをEDINETからダウンロード
   │    （lib/edinet-client.ts: downloadDocumentZip）
   │
   ├─③ ZIPを展開し、XBRL/iXBRLファイルを特定してXML解析、
   │    財務タグ（売上高・営業利益・経常利益・純利益・EPS・DPS・
   │    BPS・自己資本比率）を抽出
   │    （lib/xbrl-parser.ts: parseXbrlZip）
   │
   └─④ FinancialData として画面に返却
   │
   ▼
[画面] 財務データ行を表示 + バックグラウンドで
   GET /api/market-data?secCode=XXXX を非同期実行
   （Yahoo Financeから前日終値・業種等を取得し、後から差し込み表示）
```

### 5.2 一括処理のシーケンス

```
入力テキストを改行・カンマ・空白で分割 → 4桁数字のみ抽出（最大70件）
   │
   ▼
for (各証券コード) {
  1. status を "processing" に更新 → 画面再描画
  2. /api/financials を呼び出し（単一検索と同じ処理）
  3. 成功 → status="done" / 失敗 → status="error"（個別企業の
     失敗は他企業の処理を止めない）
  4. /api/market-data をバックグラウンドで呼び出し
  5. 600ms 待機（EDINET APIのレート制限回避）
}
```

### 5.3 期間指定（複数年度）取得のシーケンス

```
GET /api/financials?secCode=XXXX&fromYear=YYYY&toYear=YYYY（最大10年）
   │
   ▼
lib/edinet-client.ts: findDocumentsByYearRange
   ① 直近の最新1件を取得し、提出月のパターン（何月に提出する企業か）を把握
   ② 対象年ごとに「前年の提出月±45日」を5日おきにサンプリングして書類を検索
      （全年度を並列フェッチ）
   ③ docID単位で重複除去し、periodEnd（決算期末日）降順でソート
   ④ 各書類につき最大10件までZIPダウンロード＋XBRL解析を実行
   │
   ▼
複数年分の FinancialData 配列を返却（画面では複数行として表示）
```

## 6. 財務データ抽出ロジック（XBRL解析）

### 6.1 会計基準の判定

XBRLファイル内のタクソノミ名前空間プレフィックスのみで判定する（本文中の「IFRS」という単語の有無では判定しない。栄電子のようにIFRS言及があってもJGAAP企業である例があり、文字列マッチでは誤判定するため）。

| プレフィックス | 判定結果 |
|---|---|
| `jpigp` | IFRS |
| `jpus` | US_GAAP |
| `jppfs` または `jpdei` | JGAAP |
| 上記いずれも無し | Unknown |

### 6.2 財務タグの優先順位（フォールバック方式）

会計基準ごとに複数のXBRLタグ候補を優先順位付きリストで保持し、先頭から探索して最初に見つかった値を採用する（`lib/xbrl-parser.ts`）。

例：純利益（JGAAP）
1. `ProfitLossAttributableToOwnersOfParent`（親会社株主に帰属する純利益。EPSの算出基準であり、日本の一般的な「純利益」はこちら）
2. `ProfitLoss`（非支配株主分を含む連結純利益。①が見つからない場合のみ使用）
3. その他のフォールバックタグ

> 注意点（過去に発見した不整合）：①を②より後に置くと、りそなHD・SMFGなどの非支配株主が存在する企業で純利益が実際より大きい誤った値になる。

BPS（1株当たり純資産額）・自己資本比率は、IFRS企業のタクソノミ上の既知の命名不整合（`EquityToAssetRatioIFRSSummaryOfBusinessResults` というタグ名に実際はBPSの値が入っている等）に対応するため専用のタグリストを使用する。

### 6.3 連結/単体の判定とコンテキスト一致

XBRLでは同じタグ名でも「連結」「単体（提出会社のみ）」で別の`contextRef`（コンテキスト参照）を使う。

- 連結: 素のコンテキスト（例 `CurrentYearDuration`）
- 単体: `_NonConsolidatedMember` サフィックス付き（例 `CurrentYearDuration_NonConsolidatedMember`）

**完全一致（`===`）でのみ判定する。** 部分一致(`includes`)を使うと `"CurrentYearDuration_NonConsolidatedMember".includes("CurrentYearDuration")` が `true` になってしまい、単体の値を連結の値と誤認する不具合が過去に発生したため。

連結企業かどうかは「`_NonConsolidatedMember`専用コンテキストが別途存在するか」で判定する（素の`CurrentYearDuration`だけでは単体のみの企業でも使われるため判定材料にならない）。

優先コンテキストが見つからない場合は、もう一方の基準へ必ずフォールバックする（横田製作所のように単体コンテキストのみで全データを開示する企業に対応するため）。

### 6.4 ZIP内ファイルの探索順序

1. 従来型XBRL（`.xbrl`拡張子）
2. インラインXBRL（`-ixbrl.htm`）
3. `PublicDoc`ディレクトリ内のhtmlファイル
4. 任意のhtmlファイル（最終フォールバック）

マニフェスト・計算リンクベース・定義リンクベース等（`manifest`/`_cal.`/`_def.`/`_lab.`/`_pre.`/`_ref.`を含むファイル名）は除外する。

## 7. 市場データ取得ロジック（`app/api/market-data/route.ts`）

1. Yahoo Finance の非公開エンドポイント（`fc.yahoo.com`でcookie取得→`query2.finance.yahoo.com/v1/test/getcrumb`でcrumb取得）を使い認証情報を1時間キャッシュ。
2. `query1.finance.yahoo.com/v7/finance/quote` に証券コード+`.T`（東証）で問い合わせ、以下を取得：
   - `regularMarketPreviousClose`（前日終値。ライブ価格ではなくこちらを採用 — 画面表記「前日終値」と一致させるため）
   - `industry`（無ければ`assetProfile`サブAPIから取得）
   - `epsTrailingTwelveMonths` / `bookValue` / `trailingAnnualDividendRate`（Yahoo自身のTTM基準の参考値。「Yahoo TTM参考」として小さく併記表示するのみで、メインのPER/PBR/配当利回り計算には使わない）

## 8. PER・PBR・配当利回りの計算ロジック

メイン表示の指標は **EDINETの実績データ × 直前の株価** で自前計算する（Yahoo自身が持つPER/PBRはそもそも返却しない設計）。`app/page.tsx`の`exportRows`、`components/MarketDataTable.tsx`の`ratio()`関数の2箇所で同一ロジックを実装している。

| 指標 | 計算式 | 備考 |
|---|---|---|
| PER（倍） | 前日終値 ÷ EPS（実績） | EPS≤0の場合は算出せず「—」表示 |
| PBR（倍） | 前日終値 ÷ BPS（実績） | BPS≤0の場合は算出せず「—」表示 |
| 配当利回り（%） | DPS（実績） ÷ 前日終値 × 100 | DPS<0はあり得ないため除外、DPS=0は0%として表示 |

> 「Yahoo Japan等のサイト上の数値と異なる」理由：Yahoo Japanのウェブサイトは会社が公表した**業績予想（会社予想）**ベースのEPSを使ってPERを表示しているが、本アプリが利用しているYahoo Finance Global APIにはその「会社予想」相当のフィールドが存在しない（TTM実績か、アナリスト合意の予想のみ）。これは構造的なデータソースの差であり、本アプリのバグではない（画面下部の注記に明記）。

## 9. 理論株価の計算ロジック（`lib/theoretical-price.ts`）

### 9.1 計算式

```
発行済株式数（推計） = 純利益（実績） ÷ EPS（実績）
計算用EPS            = 経常利益（実績） × 0.7 ÷ 発行済株式数（推計）
ROA                  = 計算用EPS ÷ (BPS ÷ 自己資本比率)
財務レバレッジ補正    = 自己資本比率に応じた係数（下表）
割引評価率           = 自己資本比率に応じた係数（下表）
事業価値             = 計算用EPS × 15 × ROA × 10 × 財務レバレッジ補正
資産価値             = BPS × 割引評価率
理論株価             = 事業価値 + 資産価値
```

財務レバレッジ補正・割引評価率は自己資本比率に応じた段階的な係数：

| 自己資本比率 | 財務レバレッジ補正 | 割引評価率 |
|---|---|---|
| 80%以上 | 1.0 | 0.8 |
| 67%以上 | 1.0 | 0.75 |
| 66.7%以上 | 1.0 | - |
| 50%以上 | 1.25 | 0.7 |
| 33.4%以上 | 1.25 | - |
| 33%以上 | 1.5 | 0.65 |
| 10%以上 | 1.5 | 0.6 |
| 10%未満 | 1.5 | 0.5 |

> 決算短信の「予想経常利益」はEDINET経由では取得不可能（EDINETには有価証券報告書の実績データしかなく、決算短信は別システムTDnet管轄。docTypeCode=140は四半期報告書であり決算短信ではない。2024年度以降は四半期報告書自体が廃止されている）。そのため**有価証券報告書の実績経常利益**を代用し、参考値として明示している。

### 9.2 部分算出の方針（重要な設計判断）

入力データの一部が欠けている場合でも、**依存関係上算出可能な項目だけは算出して返す**（全項目を一律nullにしない）。

| 必要な入力 | 算出できる項目 |
|---|---|
| BPS・自己資本比率のみ | 自己資本比率(%)・財務レバレッジ補正・割引評価率・資産価値 |
| 上記 + 純利益・EPS | + 発行済株式数（推計） |
| 上記 + 経常利益 | + 計算用EPS・ROA・事業価値・理論株価 |

これは特にIFRS採用企業（「経常利益」という概念自体が存在しない）で重要となる。経常利益が無い企業でも、BPS・自己資本比率さえ取得できていれば残りの項目は表示される。

ガード条件（`NULL_RESULT`を返すケース）:
- BPS が `null` または `0`以下
- 自己資本比率が `null` または、% 換算後 `0`以下

## 10. データモデル（`types/financial.ts`）

```ts
// 1社・1期分の財務データ（EDINETから抽出した実績値）
interface FinancialData {
  secCode: string;              // 証券コード(4桁)
  companyName: string;
  edinetCode: string;
  docID: string;                 // EDINET書類管理番号
  fiscalYear: string;
  periodEnd: string;             // 決算期末日
  submitDateTime: string;        // 書類提出日時
  accountingStandard: "JGAAP" | "IFRS" | "US_GAAP" | "Unknown";
  isConsolidated: boolean;
  netSales: number | null;
  operatingIncome: number | null;
  ordinaryIncome: number | null; // IFRS企業はnull
  netIncome: number | null;
  eps: number | null;
  dps: number | null;
  bps: number | null;
  equityRatio: number | null;
}

// 1社分の市場データ(Yahoo Finance由来)
interface MarketData {
  secCode: string;
  currentPrice: number | null;   // 前日終値
  industry: string | null;       // JPX33業種区分（日本語）
  yahooEps: number | null;       // Yahoo自身のTTM EPS(参考表示専用)
  yahooBps: number | null;
  yahooDps: number | null;
}

// エクスポート(CSV/Excel/Sheets)1行分
interface ExportRow {
  // FinancialData・MarketData・理論株価の計算結果を1行にフラット化した構造
  // (証券コード・銘柄名・業種・前日終値・PER・PBR・配当利回り・決算期・
  //  売上高・経常利益・純利益・EPS・DPS・発表日・理論株価計算過程一式)
}
```

## 11. 出力（エクスポート）仕様

### 11.1 CSV / Excel（`lib/exporter.ts`, `app/api/export/route.ts`）

25列構成。前半14列が基本財務指標、後半11列が理論株価の計算過程（BPS・自己資本比率・株式数推計・計算用EPS・ROA・財務レバレッジ補正・割引評価率・事業価値・資産価値・理論株価）。

- CSV: UTF-8 BOM付き（Excelで文字化けしないように）
- Excel: ヘッダー行を紺色背景＋白文字太字、数値列は右揃え＋桁区切り、全セルに枠線

### 11.2 Googleスプレッドシート（`app/api/sheets-create/route.ts`）

- サービスアカウント（JWT認証、`GOOGLE_SERVICE_ACCOUNT_JSON`環境変数）で既存スプレッドシート（`GOOGLE_SPREADSHEET_ID`）に新規シートタブを追加
- ヘッダー：主要列(0-13列)は紺色、理論株価計算列(14-24列)はアンバー色で視覚的に区別
- 理論株価計算列のデータ行に薄い黄色背景＋セクション区切りの太線
- 1行目固定（フリーズ）

## 12. 既知の制約・構造的な限界

| 項目 | 内容 |
|---|---|
| 決算短信（予想値）は取得不可 | EDINETには実績データ（有価証券報告書）のみ存在し、会社予想は別システム(TDnet)管轄。docTypeCode=140(四半期報告書)は2024年度以降廃止され決算短信とは無関係。 |
| Yahoo Japan表示PERとの不一致 | Yahoo Japanサイトは会社予想EPS基準、本アプリはEDINET実績EPS基準。算出基準が異なるため一致しない（バグではない）。 |
| IFRS企業は理論株価が出ない | IFRSには「経常利益」の概念が無いため計算用EPS・ROA・事業価値・理論株価そのものは算出不可（— 表示）。自己資本比率・株式数推計・資産価値等は算出可能（[9.2](#92-部分算出の方針重要な設計判断)参照）。 |
| Yahoo Finance APIは非公式 | 公開ドキュメントの無い内部APIのため、Yahoo側の仕様変更で予告なく挙動が変わるリスクがある。 |
| 一括処理は最大70社 | Vercelサーバーレス関数のタイムアウト・EDINET APIのレート制限を考慮した上限。 |
